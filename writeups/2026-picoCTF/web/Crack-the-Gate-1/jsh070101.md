A common trick is to rotate each letter by 13 positions in the alphabet.
-> ROT13쓰라는 소리 (알파벳을 13칸 회전시키는 아주 유명한 암호 방식) 

따라서 개발자 도구와 페이지 소스를 봤는데 아래와 같은 !-- 화면에는 안보이지만 소스코드에는 남아있는 부분을 체크하여 아래와 같은 부분을 찾아냈고 이를 rot13을 이용해 해석하고자 했다.

rot13으로 해석하면 NOTE: Make - temporary bypass: use header "X-Dev-Access: yes" 이런 결과가 나와 X-Dev-Access: yes 이러한 헤더를 넣으면 우회가 가능함을 알게 되었다.

취약점 : Authentication Bypass (인증 우회) - 그중에서도 사용자가 조작 가능한 헤더를 신뢰해서 인증을 우회한것. 원래는 비번이나 세션, 토큰과 같이 서버가 검증 가능한걸 사용해야하는데 그러지 않음.

fetch('/login', { 
    method: 'POST',
    headers: {
        'Content-Type': 'application/json',
        'X-Dev-Access': 'yes'
    },
    body: JSON.stringify({
        email: 'ctf-player@picoctf.org',
        password: 'test'
    })
})
.then(res => res.json())
.then(data => console.log(data))

1. 서버한테 말걸기 fetch 함수
2. 로그인 주소로 요청 보냄
3. post : 서버에 데이터 보내는 방식
4. 헤더 시작하고 json 형식으로 데이터 보냄
5. 실제 로그인 정보 body로 실제로 데이터 보내고 자바스크립트를 json으로 바꿔주는거 씀
6. 아이디는 정해준걸로 하고 비번은 헤더 때문에 아무거나 해도 되므로 test로 설정
7. 마지막은 josn으로 응답을 전환함.

이런 코드를 콘솔 창에 넣어  X-Dev-Access: yes 이걸 활용해 개발자 권한 있어요 라고 특별 헤더를 추가해 서버를 속이고 비번은 아무거나 상관없으므로 저런식으로 코드를 짰다.

## 배운 점

이번 문제를 통해 웹 페이지의 화면만 보는 것이 아니라 페이지 소스와 개발자 도구를 확인하는 것이 중요하다는 것을 배웠다. 특히 HTML 주석이나 JavaScript 코드 내부에 개발자가 실수로 남겨둔 정보가 있을 수 있다는 점을 알게 되었다. 또한 사용자가 HTTP 요청의 헤더를 직접 수정할 수 있다는 점과 서버가 이를 신뢰하면 인증 우회(Authentication Bypass) 취약점이 발생할 수 있다는 것도 이해하게 되었다. 추가로 ROT13 같은 간단한 인코딩 방식도 CTF에서 자주 사용된다는 것을 배웠다.
