# 문제명

off_by_one
## 문제 설명

- 본 문제는 사용자로부터 이름과 나이를 입력받아 처리하는 단순한 구조의 프로그램에서 발생하는 취약점을 다룹니다. 프로그램 내부에는 나이(age) 변수가 0일 경우 관리자 권한을 부여하고 쉘을 실행해 주는 get_shell() 함수가 분기문으로 존재합니다. 하지만 정상적인 입력 흐름에서는 나이를 0으로 설정할 수 없도록 방어 로직이 설계되어 있습니다. 개발자는 문자열을 안전하게 입력받기 위해 버퍼의 크기만큼 데이터를 읽어온 뒤 맨 뒤에 Null 바이트(\x00)를 명시적으로 삽입해 주는 사용자 정의 함수(read_str)를 사용합니다. 그러나 이 함수 내부에서 문자열의 끝을 지정하는 인덱스 계산에 미세한 오차가 존재하며, 이로 인해 입력 버퍼의 크기를 가득 채웠을 때 범위를 정확히 1바이트 벗어나 뒤에 인접한 변수를 오염시킬 수 있는 구조적 결함을 가지고 있습니다. 즉, 메모리 경계 검사의 허점을 이용해 데이터의 흐름을 조작하고 프로그램이 의도하지 않은 인증 우회를 달성하는 것이 이 문제의 핵심 목표입니다.
 

## 풀이

### 분석

문제 소스 코드에서는 사용자의 이름과 나이를 입력받는 전역 변수 또는 스택 변수가 존재하며, 나이가 0일 때 프로그램 내부에서 'get_shell()' 함수를 호출하여 권한을 부여하도록 설계되어 있습니다. 사용자로부터 안전하게 문자열을 입력받기 위해 개발자는 'read_str'이라는 별도의 함수를 정의하여 사용합니다. 이 함수는 지정된 크기만큼 입력을 받은 뒤, 입력받은 문자열의 바로 다음 인덱스 위치에 문자열의 끝을 알리는 Null 바이트(\x00)를 강제로 삽입하는 구조를 가지고 있습니다. 메인 루틴에서는 이름 배열의 크기로 지정된 버퍼 크기를 인자로 넘겨주며 입력을 시도하고, 만약 나이가 0이 아니라면 다른 분기로 넘어가게 됩니다. 하지만 메모리 구조상 이름 버퍼 바로 뒤에 나이 변수(age)가 인접하여 배치되어 있기 때문에 문자열 경계 처리에 미세한 오차가 발생할 경우 바로 뒤에 있는 변수의 값에 영향을 줄 수 있는 구조입니다.

### 취약점

이 프로그램의 핵심 결함은 read_str 함수가 문자열의 끝에 Null 바이트를 삽입하는 과정에서 경계 검사를 잘못 수행하여 발생하는 Off-by-One 취약점입니다. 개발자는 버퍼의 최대 크기만큼 입력을 허용하면서, 입력받은 길이의 인덱스 위치에 Null 바이트를 넣도록 코드를 작성했습니다. 예를 들어 크기가 20인 배열에 정확히 20바이트를 꽉 채워 입력하게 되면, 배열의 인덱스는 0부터 19까지인데도 불구하고 buf[20] 위치에 Null 바이트를 쓰게 됩니다. buf[20]은 배열의 범위를 정확히 1바이트 벗어난 위치이며, 메모리 구조상 이 자리는 이름 버퍼 바로 뒤에 선언된 age 변수의 가장 낮은 자릿수 바이트(Little Endian 구조상 데이터의 시작점)와 일치하게 됩니다. 결과적으로 이름 입력란에 글자를 꽉 채워 보내는 것만으로도 의도치 않게 뒤에 있는 age 변수의 값을 0x00으로 조작할 수 있는 단 1바이트의 오버플로우가 발생합니다.

### 익스플로잇

메모리가 인접해 있다는 특성과 Off-by-One 취약점을 연계하면 복잡한 주소 연산 없이 아주 간단하게 조건문을 우회할 수 있습니다. 프로그램이 시작되면 나이를 0이 아닌 값으로 먼저 입력하여 초기화 과정을 거친 뒤, 이름 입력 메뉴로 진입합니다. 이름 입력란에 버퍼의 최대 크기인 20바이트를 한 바이트도 남기지 않고 'A'와 같은 임의의 데이터로 가득 채워 전송합니다. 이렇게 하면 read_str 함수가 동작하면서 20번째 인덱스, 즉 이름 배열의 범위를 1바이트 초과한 위치인 age 변수의 메모리 첫 바이트에 0x00 을 강제로 덮어쓰게 됩니다. 결과적으로 age 변수의 값이 완전히 0으로 변하게 되며, 이후 프로그램이 if (age == 0) 조건을 검사할 때 조건식이 참이 되어 내부에 존재하는 get_shell() 함수가 실행되고 성공적으로 쉘을 획득할 수 있습니다.
```
python

from pwn import *

p = remote("host3.dreamhack.games", 15886)

payload = b'A' * 20

p.sendline(payload)
p.interactive()
```

## 배운 점
- 이번 문제를 풀면서 단 1바이트의 경계선 계산 실수가 프로그램 전체의 흐름을 얼마나 치명적으로 바꿀 수 있는지 생생하게 배울 수 있었습니다. 인덱스가 0부터 시작한다는 기본적인 마인드를 소홀히 하거나 반복문의 조건식을 느슨하게 검사할 때 발생하는 Off-by-One 취약점이, 메모리 상에 인접한 변수를 오염시켜 인증을 우회하는 강력한 공격 수단이 될 수 있음을 깨달았습니다. 데이터를 안전하게 처리하기 위해서는 단순히 버퍼 크기만큼 읽어오는 것뿐만 아니라, Null 바이트가 삽입될 공간까지 엄격하게 계산하여 경계 범위를 설정해야 한다는 시큐어 코딩의 중요성을 깊이 절감했습니다. gdb를 통해 메모리를 들여다보며 데이터가 바이트 단위로 정렬되어 있는 모습과 Little Endian 방식으로 값이 들어가는 과정을 직접 확인하는 과정이 매우 흥미로웠습니다. 복잡한 페이로드를 구성하지 않더라도 시스템의 메커니즘을 정확히 이해하고 있다면 단 1바이트의 조작만으로도 완벽한 익스플로잇이 가능하다는 것을 알게 되었습니다. 앞으로 해킹 문제를 분석하거나 직접 개발을 진행할 때, 경계 조건 값(Boundary Value) 하나하나를 더욱 예리하고 밀도 있게 검증하는 습관을 지녀야겠다고 느꼈습니다.
