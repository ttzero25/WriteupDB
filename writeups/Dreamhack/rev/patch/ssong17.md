# 문제명

## 문제 설명

flag를 그리는 루틴을 분석하고 가려진 flag를 보이게 해주세요

- 문제 URL / 파일 등 접속 정보: patch.exe 파일

## 풀이

### 분석

우선 디버거를 켜고 `patch.exe`를 실행한 뒤 EP에 BP를 걸어 빠르게 내부로 진입했다.
 
어떤 함수를 지나고 나니 DH 관련 창이 떴고, 이 지점이 flag를 그리는 부분이라고 판단해 여기에 BP를 걸었다. 들어가서 관찰해 보니 다음 Win32 API들이 보였다.
 
- `LoadIconW`, `LoadCursorW`, `LoadStringW`
- `CreateWindowExW`
- `ShowWindow`, `UpdateWindow`

즉 아이콘/문자열을 로드한 뒤 창을 만들고, 화면을 갱신하면서 무언가를 그려내는 흐름이었다. 실행 순서를 정리하면 **아이콘 가져오기 → 창 띄우기 → 그림 그리기** 순서였다.
 
좀 더 살펴보니 위쪽에서 `patch.7FF7BDC02B80`을 계속 호출하면서 무언가를 그리고 있었다. 주변에는 다음과 같은 문자열이 주석으로 달려 있어서 일단 전부 적어두었다.

```
P#K(F-A2<77<2A-F(KKP<Z##(-22(7Z<<AFPKFP<PPPFZZ(P
```
 
그리는 함수 호출 부분(`patch.7FF7BDC02B80`)에 BP를 걸고 반복 실행하며 관찰했다. 일부만 캡쳐해 보니 이상한 그림이 그려지다가 갑자기 DH가 딱 하고 띄워졌다.
 
그래서 창이 띄워지는 부분부터 한 줄씩 따라가 보기로 했다. `UpdateWindow` 함수 안으로 들어가 천천히 추적하니, Syscall 이후 다시 `patch.7FF7BDC02B80`이 호출되며 미리 걸어둔 BP로 넘어왔다.
 
그 안에서는 `GdipCreatePen1 → GdipDrawLineI` 순서로 첫 번째 선이 그려지기 시작했고, 함수 안에서는 계속 이 GDI+ 호출만으로 선을 하나씩 이어 그리고 있었다.


### 취약점

flag는 별도의 검증/암호화 없이 단순히 GDI+ 선 그리기로 화면에 직접 그려진다. 앞쪽에서 의미 없는(난독화용) 도형을 잔뜩 그려 실제 flag를 가려두고, 그 뒤에 진짜 flag를 그리는 구조였다.
 
따라서 그리기 루틴의 흐름만 패치해서 **앞쪽 가짜 그림을 그리는 부분을 건너뛰고, 실제 flag를 그리는 부분부터 실행**되게 만들면 가려진 flag를 그대로 볼 수 있다.

### 익스플로잇

그리는 함수를 수십 번 단계 실행하며 DH(flag)가 그려지기 시작하는 지점을 찾았다. 핵심 아이디어는 맨 처음 그리기를 호출하는 부분에서, 실제 DH 시작 부분으로 바로 `jmp` 시키는 것이었다.
 
삽질을 좀 한 끝에, 맨 처음 그리기 호출 부분을 다음과 같이 패치했다.
 
```asm
; 맨 처음 그리기 호출(가짜 그림 그리는 부분)을
; DH 그리기 시작 지점으로 직접 점프
jmp 0x00007FF7BDC02FE6
```
 
이렇게 앞쪽 난독화 그림 부분을 건너뛰고 실행했더니, 가려져 있던 flag가 화면에 그려졌다.
 
다만 점프 시작 위치를 약간 잘못 짚었는지 `D` 글자가 온전히 나오지 않았다. 그래도 나머지가 정상적으로 그려져 flag는 충분히 식별할 수 있었다.
